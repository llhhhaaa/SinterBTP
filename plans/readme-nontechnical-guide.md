## 1. 许可证 (License)

### 1.1 三种推荐协议对比

| 维度 | MIT | Apache 2.0 | GPL v3 |
|:---|:---|:---|:---|
| 核心理念 | 极度宽松，几乎无限制 | 宽松 + 专利保护 | 强 Copyleft，传染性 |
| 商业使用 | ✅ 允许，无任何附加条件 | ✅ 允许，需保留声明 | ✅ 允许，但衍生作品必须同样开源 |
| 修改再分发 | ✅ 可闭源分发 | ✅ 可闭源分发，需注明修改 | ❌ 衍生作品必须以 GPL 发布 |
| 专利授权 | ❌ 无明确专利条款 | ✅ 明确授予专利使用权 | ✅ 隐含专利授权 |
| 责任担保 | 免责（AS IS） | 免责（AS IS） | 免责（AS IS） |
| 文件要求 | 仅需保留版权声明 | 需保留 NOTICE 文件 + 标注修改 | 需提供完整源码 |
| 典型用户 | 个人项目、学术项目 | 企业级开源、涉及专利 | 希望保持开源生态纯净 |

### 1.2 针对本项目的推荐

**首选：MIT License**

理由：
- 本项目是本科毕业设计，核心目标是学术传播和知识共享，而非商业保护
- MIT 协议是 GitHub 上最流行的协议，对潜在使用者的门槛最低
- 烧结领域的研究者和工程师可以自由地将代码集成到自己的系统中，有利于学术影响力扩散
- 协议文本极短（约 170 词），维护成本为零

**备选：Apache 2.0**

如果你担心未来有人基于你的模型架构申请专利反过来限制你，Apache 2.0 的专利授权条款能提供额外保护。但对于毕设项目，这种风险极低。

**不推荐 GPL v3**

GPL 的传染性会阻止企业用户将你的代码集成到闭源的工业控制系统中，这与"让更多人用到"的开源初衷相悖。
